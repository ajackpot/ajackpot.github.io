# 테스트 보고서

## 실행 방식

브라우저 자동화는 Playwright Core를 사용했습니다.

이 세션에서는 일부 관리형 Chromium 환경에서 `localhost` 또는 `file://` 로딩이 정책상 차단될 수 있어, 테스트 하네스가 앱 모듈을 **in-memory data URL** 로 재작성하여 로딩하도록 구성했습니다.

## 실행 명령

```bash
npm run test:agent
```

## 검증 항목

### 1. 초기 렌더링과 접근성 이름
- 체스판이 표로 렌더링되는지 확인
- `e2` 칸의 접근 가능한 이름이 `백 폰, e2` 인지 확인
- 초기 `aria-pressed` 값 확인
- 사용자 지정 입력의 기본 비활성화 상태 확인
- 새 게임 시작 안내가 라이브 리전에 올라오는지 확인

### 2. 키보드 탐색과 선택/이동
- 방향키로 보드 안 포커스 이동 확인
- `Enter`로 기물 선택 가능 여부 확인
- 라이브 리전에 `선택됨` 메시지가 올라오는지 확인
- `Space`로 목적지 이동 가능 여부 확인
- 이동 후 기보 갱신과 선택 해제 확인

### 3. 사용자 지정 옵션 잠금/해제
- 난이도 `사용자 지정` 선택 시 세부 입력 활성화 확인
- 프리셋으로 돌아가면 다시 비활성화되는지 확인

### 4. AI 탐색 중 포커스 보존
- AI 탐색 진행 중 상태 갱신이 보드 밖 포커스를 빼앗지 않는지 확인

### 5. 워커 탐색 취소 후 즉시 재시작
- 긴 탐색 도중 새 게임 시작 시 이전 탐색이 실제로 중단되는지 확인
- 재시작 후 AI 첫 수가 지연 없이 반영되는지 확인

## 이번 단계에서 추가로 잡은 문제

### A. 라이브 리전 테스트의 간헐적 실패
증상:
- 선택 직후 라이브 리전 텍스트를 즉시 읽으면 빈 문자열이 읽히는 경우가 있었습니다.

원인:
- `LiveRegion`이 모든 공지를 `clear -> timeout -> set` 방식으로 처리해 아주 짧은 공백 구간이 생겼습니다.

수정:
- **새 메시지가 이전 메시지와 다를 때는 즉시 반영**하도록 바꾸고,
- 같은 메시지를 연속 공지할 때만 `clear -> delayed set` 경로를 사용하도록 개선했습니다.
- 테스트도 라이브 리전 텍스트를 기다린 뒤 검증하도록 보강했습니다.

### B. 테스트 환경의 Chromium 경로 하드코딩
증상:
- 테스트 하네스가 `/usr/bin/chromium`만 가정하고 있었습니다.

수정:
- `CHROMIUM_PATH` 환경 변수와 주요 운영체제의 일반 경로 후보를 순서대로 확인하게 바꿨습니다.
