# 18단계 보고서: 결과 화면 문구, 댓글 답글 질문, 예약 완료 화면 정비

## 1. 작업 목적

이번 단계는 실제 테스트 페이지를 본 뒤 발견된 화면 흐름 문제를 정리하기 위해 진행했다. 핵심은 다음 네 가지다.

- 최종 결과 화면에서 이미 지나간 안내를 반복하지 않는다.
- 결과 파일 내려받기 안내와 기록 확인 안내의 읽기 순서를 자연스럽게 정리한다.
- 댓글 목록의 답글 보기와 도움이 돼요 동작을 일반적인 서비스 피드백에 가깝게 바꾼다.
- 예약 확정 뒤 일반 처리 완료 대화상자 대신 예약 완료 화면을 표시한다.

첨부 연구 메모의 방향처럼 이 실험은 단순히 키보드로 조작 가능한지보다 목표 도달까지의 조작 비용과 실제 수행 성과를 보려는 구조다. 따라서 수행 중에는 정답 여부를 알려 주지 않고, 최종 과업 종료 뒤에만 판정 문구를 제공한다.

## 2. 결과 화면 문구 정리

### 2.1 제거한 문구

최종 결과 화면의 `기록 확인 안내`에서 다음 문구를 제거했다.

- `두 화면을 모두 수행한 뒤에만 결과가 표시됩니다.`

이 문구는 과업 전 안내로는 필요하지만, 이미 두 화면을 모두 수행해 최종 결과 화면에 온 뒤에는 반복 안내가 된다.

### 2.2 결과 파일 안내 위치 변경

다음 문구를 결과 파일 내려받기 링크 앞에 배치했다.

- `결과 파일(JSON)을 내려받아 설문 응답과 함께 보관할 수 있습니다.`

적용 대상은 세 서비스 모두다.

- `app.js`
- `comments-app.js`
- `search-app.js`

### 2.3 기록 확인 안내 목록 정리

`과업 내용 확인 시간은 메인 창에서 분리되며, 수행 탭이 숨겨진 동안의 시간은 실제 완료 시간에서 뺍니다.` 문구와 `과업 종료 버튼을 너무 일찍 누른 기록은 수행 불가능 기록으로 표시됩니다.` 문구를 같은 `기록 확인 안내` 목록으로 묶었다.

## 3. 과업 준비 화면 문구 수정

다음 문구가 어색하다는 지적을 반영했다.

- 기존: `수행할 수 없다고 판단해도 수행 탭의 과업 종료 버튼으로 다음 단계로 넘어갑니다.`
- 변경: `수행할 수 없다고 판단해도 최하단의 과업 종료 버튼을 실행하여 다음 단계로 건너뛸 수 있습니다.`

적용 대상은 다음 파일이다.

- `lib/service-shell.js`
- `app.js`
- `comments-app.js`
- `search-app.js`
- `docs/manual-release-checklist.md`

## 4. 댓글 목록 상호작용 변경

### 4.1 답글 보기 흐름 변경

기존에는 `답글 보기`를 실행하면 `처리가 완료되었습니다.` 대화상자가 열렸다. 이번 단계에서는 이를 제거하고 실제 답글 목록이 열리도록 바꿨다.

답글 목록에는 각 답글의 다음 정보가 표시된다.

- 몇 번째 답글인지
- 작성자
- 작성 시간
- 답글 본문

### 4.2 객관식 답글 질문 추가

답글 목록이 열린 뒤, 과업에서 요구하는 답글 정보를 콤보상자로 선택해 제출하도록 바꿨다.

현재 댓글 목록 과업은 다음처럼 바뀌었다.

- 과업 1: 민지 댓글의 답글 목록에서 두 번째 답글 작성자를 선택해 제출
- 과업 2: 운영자 안내 댓글 정보 보기 후 도움이 돼요 표시
- 과업 3: 준호 댓글의 답글 목록에서 세 번째 답글 작성자를 선택해 제출

객관식 선택지는 5개로 구성했다. 주관식 입력은 오타나 띄어쓰기 논쟁이 생길 수 있어 사용하지 않았다.

### 4.3 판정 방식

`replyQuestion` 과업 판정 조건을 추가했다.

성공으로 기록되려면 다음 조건을 만족해야 한다.

- 필요한 정렬/범위 조건을 맞춘다.
- 요청한 댓글의 답글 목록에 진입한다.
- 요청한 답글 질문에 맞는 선택지를 제출한다.
- 다른 댓글의 답글 질문에 답을 제출하지 않는다.

요청한 답글 목록에 진입하지 않았거나, 다른 댓글의 답글 질문에 답을 제출했거나, 답을 제출하지 않았거나, 다른 답을 제출한 경우에는 과업 종료 뒤 실패 사유로 표시된다. 수행 중에는 이 사유를 알려 주지 않는다.

### 4.4 도움이 돼요 흐름 변경

`도움이 돼요` 버튼은 더 이상 처리 완료 대화상자를 열지 않는다.

대신 화면 하단에 다음과 같은 토스트 알림을 표시한다.

- `도움이 돼요를 표시했습니다.`
- 이미 표시된 경우: `이미 도움이 돼요가 표시되어 있습니다.`

토스트는 `role="status"`, `aria-live="polite"` 구조를 사용해 화면낭독기가 읽을 수 있다. 관련 없는 댓글에서 눌러도 같은 종류의 토스트가 표시되지만, 최종 과업 성공으로는 기록하지 않는다.

## 5. 예약 캘린더 상호작용 변경

### 5.1 예약 확정 뒤 흐름 변경

기존에는 예약 확정 뒤 `처리가 완료되었습니다.` 대화상자가 열렸다. 이번 단계에서는 이를 제거하고 서비스 화면 안에 `예약 완료 화면`을 표시한다.

예약 완료 화면에는 다음 정보가 들어간다.

- `예약이 완료되었습니다` 제목
- 예약한 시간 요약
- 예약 내용은 `현재 예약 내용` 영역에서 확인할 수 있고, 필요한 경우 그 영역에서 취소할 수 있다는 안내

### 5.2 과업 판정 변경

예약 캘린더 과업은 더 이상 `최종 확인 대화상자 확인 여부`를 성공 조건으로 쓰지 않는다.

성공으로 기록되려면 다음 조건을 만족해야 한다.

- 요청한 예약 시간으로 예약이 저장된다.
- 예약 완료 화면에 진입한다.
- 취소 후 재예약 과업에서는 기존 예약 취소가 실제로 기록된다.

요청한 시간과 다른 시간을 예약하면 `요청한 상담 예약 시간과 다른 시간을 예약했습니다.`로 기록한다. 예약 완료 화면에 이르지 못하면 `예약 확정 화면에 진입하지 못했습니다.`로 기록한다.

## 6. 내부 상태와 기록 항목

댓글 목록에 다음 내부 상태를 추가했다.

- `replyListVisitedThisTask`
- `replyAnswerDrafts`
- `submittedReplyAnswers`

예약 캘린더에 다음 내부 상태를 추가했다.

- `bookingCompletion`

이 값들은 화면 판정과 결과 JSON에는 사용되지만, 실험 번호와 마찬가지로 사용자에게 불필요한 내부 식별값으로 직접 노출하지 않는다.

- 댓글 답글 목록 상태에 `처리가 완료되었습니다.` 일반 대화상자 문구가 나오지 않는지

## 8. 검증

다음 명령을 실행했다.

```bash
node --check app.js
node --check comments-app.js
node --check search-app.js
node --check data/tasks-comments.js
node --check data/comments-scenario.js
node --check data/benchmark-graphs-comments.js
node --check scripts/smoke-render-runner.mjs
node scripts/run-benchmark.mjs
npm run check
```

사전 계산 결과는 다시 생성했다.

- `data/benchmark-results-calendar.json`
- `data/benchmark-results-comments.json`
- `data/benchmark-results-search.json`

댓글 목록은 답글 질문 제출 단계가 추가되어 예상 시간이 갱신되었다.

- 키보드 사용자: A 110.2초 → B 41.9초
- 화면낭독 사용자: A 189.2초 → B 69.1초
- 스위치 사용자: A 269.3초 → B 96.7초

## 9. 남은 수동 점검

브라우저와 보조기술 조합에서 다음 항목은 수동으로 확인해야 한다.

- 결과 파일 안내 문구가 내려받기 링크보다 먼저 읽히는지
- 댓글 답글 목록에서 콤보상자 선택과 제출이 실제 키보드/화면낭독기 환경에서 자연스러운지
- 도움이 돼요 토스트가 화면 하단에 표시되고 화면낭독기가 읽는지
- 예약 확정 뒤 예약 완료 화면으로 초점이 이동하는지
- 예약 완료 화면을 본 뒤 현재 예약 내용 영역에서 취소할 수 있는지
- 최종 과업 결과는 `과업 종료` → `예, 종료합니다` 뒤에만 나오는지
