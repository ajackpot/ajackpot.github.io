# 4단계 보고서 · 메인 창/수행 탭 분리와 계측 규칙 정비

## 이번 단계 목표

사용자가 지적한 두 가지 문제를 함께 해결하는 것이 목표였습니다.

1. 실제 완료 시간이 과업 설명을 읽는 시간까지 함께 재고 있었다.
2. 과업 도중 설명을 다시 확인하려면 같은 화면 안에서 과업 안내와 A/B 페이지를 오가야 해서 노이즈가 생겼다.

이를 해결하기 위해 **메인 창은 과업 안내와 진행 관리**, **새 탭은 실제 A/B 수행 화면**으로 역할을 분리했다.

---

## 구현한 내용

### 1) 메인 창에서 과업 내용을 먼저 확인하도록 흐름 변경

이제 서비스 화면에서 바로 과업을 시작하지 않는다.

새 흐름은 다음과 같다.

- 홈 화면
- 서비스 유형 선택
- 서비스 소개 화면
- 과업 준비 화면
- 새 탭에서 수행 화면 열기
- 메인 창에서 결과 수신 후 과업 결과 요약

과업 준비 화면에는 아래 정보가 유지된다.

- 현재 비교안과 과업 번호
- 과업 목표 요약
- 단계별 수행 지시
- 현재 예약 상태
- 목표 예약 시간
- 실제 계측 규칙
- 사전 계산 기준 표

즉, 사용자는 **과업 시작 버튼을 누르기 전에 수행 내용을 충분히 읽을 수 있고**, 수행 중에도 메인 창으로 돌아와 다시 확인할 수 있다.

### 2) 수행 화면을 별도 탭으로 분리

정적 앱 구조를 유지하기 위해 별도 HTML 파일을 늘리지 않고, 같은 `index.html`을 쿼리 문자열로 분기하는 방식으로 구성했다.

- 기본 모드: 메인 창
- `?mode=runner`: 수행 탭

수행 탭은 다음 원칙으로 렌더링한다.

- 실험 운영용 요약 카드, 과업 안내 카드, 용어 설명 카드는 출력하지 않음
- A/B 서비스 화면 콘텐츠만 출력
- 맨 아래 보조 영역에는 아래 두 버튼만 둠
  - 결과로 바로 이동
  - 이 탭 닫기

이렇게 해서 수행 탭에서는 불필요한 부가 정보가 빠지고, 실제 서비스 조작에 가까운 화면만 남도록 했다.

### 3) 실제 계측 시작 시점 변경

기존에는 과업 화면이 열리는 순간부터 시간이 흘렀다.

이번 단계에서는 기록 수집기를 고쳐서 다음 규칙을 적용했다.

- 기록 수집기는 수행 탭이 열릴 때 생성하되,
- **실제 계측 시작 시점은 첫 키 입력 또는 첫 클릭이 들어간 순간**으로 바꿈

따라서 사용자가 메인 창에서 과업 내용을 읽는 시간, 새 탭으로 이동하기 전 준비 시간은 실제 완료 시간에 포함되지 않는다.

### 4) 수행 탭이 숨겨져 있는 시간 제외

사용자가 과업 도중 메인 창으로 돌아가 과업 설명을 다시 확인할 수 있도록 요구했기 때문에, 단순히 별도 탭만 여는 것으로는 부족했다.

그래서 기록 수집기에 `visibilitychange` 처리를 추가했다.

- 수행 탭이 숨겨지면 일시 중지
- 다시 보이면 숨겨져 있던 시간을 누적
- 최종 완료 시간에서 누적 숨김 시간을 제외

이제 결과 요약에는 아래 두 값이 함께 남는다.

- 실제 완료 시간
- 숨김 탭 제외 시간

### 5) 수행 탭 아래 보조 버튼 제외 처리

수행 탭 맨 아래의 두 보조 버튼은 실험 운영 편의를 위한 장치이므로, 실제 서비스 조작 지표에 섞이지 않도록 별도 예외 처리했다.

- `data-measurement-exempt="true"` 표식 추가
- 기록 수집기가 해당 영역의 클릭/초점 이동을 무시

탭 이동 자체까지 완전히 제거하지는 않았지만, 보조 버튼을 클릭한 행위가 실제 서비스 조작 수치에 직접 더해지지 않도록 정리했다.

### 6) 메인 창과 수행 탭 사이 통신 추가

정적 앱이므로 서버 없이 같은 출처 통신만 사용했다.

- 기본: `BroadcastChannel`
- 대체 경로: `localStorage` 기반 저장 이벤트

메인 창과 수행 탭은 아래 상태를 주고받는다.

- 수행 탭 준비 완료
- 실제 계측 시작
- 과업 완료 결과
- 수행 탭 닫힘

또한 수행 탭이 새로 열릴 때 필요한 현재 상태(필터, 현재 예약, 현재 비교안, 현재 과업 번호)는 `localStorage`에 임시로 저장한 뒤 수행 탭이 읽도록 했다.

### 7) 결과 및 내보내기 정보 보강

다음 항목을 실제 기록과 내보내기 데이터에 추가했다.

- `hiddenDurationSeconds`
- `measurementRules`
- 계측 방식 설명 메모

또한 사전 계산 결과 생성 스크립트를 다시 실행해 최신 시각으로 결과 파일을 갱신했다.

## 검증 내용

이번 단계에서 확인한 항목은 다음과 같다.

- `node --check app.js`
- `node --check lib/logger.js`
- `node scripts/run-benchmark.mjs`
- 정적 서버 응답 확인 (`python -m http.server` + `curl`)

브라우저 자동화 기반의 끝단 E2E 주행은 이 단계에서 완료하지 못했다. 다음 단계에서는 실제 로컬 브라우저에서 아래 항목을 수동 점검하는 것이 좋다.

- 새 탭 열기 차단 여부와 안내 문구
- 메인 창에서 설명 확인 → 수행 탭 시작 → 다시 메인 창 확인 흐름
- 숨김 탭 시간이 실제로 제외되는지
- 과업 완료 후 메인 창 결과 요약 수신 여부
- 화면낭독기에서 수행 탭 하단 버튼과 대화상자 동작

## 남은 과제

1. 수행 탭을 닫지 않고도 메인 창으로 쉽게 복귀하는 사용 흐름 문구를 더 다듬기
2. 보조 버튼 영역의 탭 이동까지 지표에서 완전히 뺄지 결정하기
3. 실제 브라우저 수동 점검과 화면낭독기 점검 기록 남기기
4. 같은 운영 구조를 댓글 목록/상품 옵션 선택에도 재사용하기

## 요약

이번 단계에서 예약 캘린더 실험은 다음 구조로 정리되었다.

- **메인 창**: 과업 이해와 진행 관리
- **수행 탭**: 실제 A/B 서비스 조작
- **기록 수집기**: 첫 입력부터 시작, 숨김 시간 제외

즉, 사용자가 지적한 “과업 내용 확인 시간으로 인한 노이즈” 문제를 정면으로 줄이는 쪽으로 실험 구조를 한 단계 더 정비했다.
