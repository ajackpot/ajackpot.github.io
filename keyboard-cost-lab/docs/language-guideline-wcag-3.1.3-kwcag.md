# 화면 문구 작성 가이드

부제: WCAG 2.2 SC 3.1.3과 한국형 웹 접근성 용어를 반영한 실험 화면 문구 기준

## 1. 목적

이 프로젝트의 사용자는 과도한 키보드 조작을 테스트하러 오는 것이지, 낯선 영어 표현이나 내부 연구 용어를 학습하러 오는 것이 아니다. 화면 문구는 다음 순서로 정한다.

1. 먼저 이해하기 쉬운 한국어를 쓴다.
2. 의미 손실이 크지 않으면 널리 쓰이는 한국어식 외래어로 바꾼다.
3. 대체하기 어려운 용어는 그대로 두되, 처음 나올 때 짧은 설명을 함께 둔다.
4. 반복해서 쓰는 핵심 용어는 한 번 정하면 화면 전반에 같은 뜻으로만 쓴다.
5. 과업 수행 전에는 테스트 결과에 영향을 줄 수 있는 사전 예상 시간, 개선 목표, 조작 전략을 보여 주지 않는다.

## 2. 적용 근거

### 2.1 WCAG 2.2 SC 3.1.3

- 낯설거나 제한된 의미로 쓰는 단어, 관용 표현, 전문 용어에는 뜻을 알 수 있는 방법을 제공해야 한다.
- 기술 용어나 특수한 뜻의 단어를 쓰더라도 사용자가 그 뜻을 바로 확인할 수 있어야 한다.

### 2.2 한국형 웹 접근성 관련 용어 우선 사용

공공 웹 접근성 관련 기준과 품질 진단 기준에서 자주 쓰는 표현을 우선 채택한다.

- 키보드 사용 보장
- 초점 이동
- 반복영역 건너뛰기
- 적절한 링크 텍스트
- 레이블 제공

또한 탐색 영역의 링크명은 이해하기 쉬운 용어를 사용하도록 정한다.

## 3. 기본 작성 원칙

### 3.1 금지 또는 지양

- 화면 이해에 꼭 필요하지 않은 영어 표현
- 내부 구현 용어를 그대로 드러내는 표현
- 의미 설명 없이 쓰는 짧은 약어
- 같은 뜻인데 화면마다 다른 표현
- 과업 수행 방법을 지나치게 구체적으로 알려 주는 표현
- 사전 계산 시간, 목표 개선폭, 사용자 유형별 기준처럼 사용자의 행동을 특정 해석 틀에 가둘 수 있는 표현

예시:

- `slot` → 지양
- `skip link` → 지양
- `modal escape` → 지양
- `wrong selection` → 지양
- `pointer activation` → 지양
- `A안은 조작 부담이 크고 B안은 개선안입니다` → 과업 수행 전에는 지양
- `몇 번 안에 목표에 도달해야 합니다` → 과업 수행 전에는 지양

### 3.2 권장 방식

- 동작과 대상을 함께 드러내는 문구를 쓴다.
- 버튼과 링크는 눌렀을 때 일어나는 결과를 바로 알 수 있게 쓴다.
- 조건, 결과, 예약 내용처럼 화면의 역할이 드러나는 제목을 쓴다.
- 키 이름은 가능한 한 한국어식 표기를 함께 쓴다.
- 같은 화면에 비슷한 동작의 버튼이 여러 개 있으면 대상 이름을 함께 넣어 구분한다.
- 과업 준비 화면에서는 사용자가 달성해야 할 요청만 간단히 제시한다.
- 과업 수행을 끝냈다고 판단하면 `과업 종료` 버튼을 누르라는 안내는 반드시 제공한다.

예시:

- `결과로 바로 이동`
- `현재 예약 취소`
- `예약 가능한 시간`
- `조건 적용`
- `과업 종료`
- `결과 파일(JSON) 내려받기`

## 4. 표준 용어 대응표

| 내부/연구 용어 | 화면에 쓰는 말 | 필요 시 설명 |
|---|---|---|
| slot | 예약 시간 / 예약 가능한 시간 | 시간 한 칸을 뜻함 |
| skip link | 결과로 바로 이동 | 문서에서는 `반복영역 건너뛰기 링크`로 기록 가능 |
| modal dialog | 대화상자 | 확인이나 취소를 위해 잠깐 뜨는 작은 창 |
| focus | 초점 | 키보드로 현재 선택된 위치 |
| benchmark | 사전 계산 기준 | 실제 실험 전에 계산해 둔 예상 조작 부담. 과업 수행 전에는 노출하지 않음 |
| profile | 사용자 유형 | 키보드 / 화면낭독 / 스위치 |
| wrong selection | 목표와 다른 선택 | 요청과 다르게 고른 횟수 |
| context reset | 위치 다시 찾기 | 대화상자를 닫거나 확정한 뒤 원래 자리 대신 제목 근처로 돌아가 다시 찾아야 하는 경우 |
| modal escape | 대화상자 바깥으로 초점 이탈 | 대화상자가 열려 있는데 초점이 밖으로 빠진 경우 |
| pointer activation | 클릭 입력 | 마우스나 유사 클릭 입력 |
| task end | 과업 종료 | 사용자가 해당 화면 수행을 마쳤거나 수행할 수 없다고 판단해 기록을 저장하는 동작 |
| incomplete / unable | 수행 불가능 기록 | 목표 행동을 끝내지 못한 상태로 과업 종료를 누른 기록 |
| reply | 답글 | 기존 댓글 아래에 이어지는 댓글 |
| reply question | 답글 목록 질문 | 답글 목록을 본 뒤 특정 답글의 작성자나 작성 시간처럼 객관식으로 제출하는 확인 질문 |
| toast | 토스트 알림 | 화면 하단에 잠시 표시되는 상태 안내. 화면낭독기가 읽을 수 있도록 상태 메시지로 제공 |
| booking complete view | 예약 완료 화면 | 예약 확정 뒤 서비스 화면 안에 표시되는 완료 정보 영역 |
| sort | 정렬 기준 | 댓글이나 목록을 어떤 순서로 볼지 정하는 기준 |
| search result | 검색 결과 | 검색어에 맞아 화면에 표시된 자료 목록 |
| preview | 미리보기 | 자료를 바로 열기 전에 핵심 내용만 잠깐 확인하는 작은 창 |
| save | 저장 | 나중에 다시 보려고 자료를 보관 목록에 넣는 기능 |
| relevance | 정확도순 | 검색어와 더 가깝다고 판단한 자료부터 보는 정렬 기준 |
| JSON | 결과 파일(JSON) | 텍스트 기반 파일 형식 이름 |

## 5. 화면별 규칙

### 5.1 홈 화면

- 바로 `실험 시작` 버튼을 두지 않는다.
- 먼저 서비스 유형을 고르게 한다.
- 공개 서비스는 현재 실험 범위인 `예약 캘린더`, `댓글 목록`, `검색 결과 목록` 3개만 표시한다.
- 서비스 카드의 버튼은 `예약 캘린더 화면으로 이동`처럼 대상 이름과 결과를 함께 드러내는 문구를 쓴다.

### 5.2 서비스 화면

- `과업 준비로 이동`과 `서비스 선택으로 돌아가기`를 함께 둔다.
- 사용자가 무엇을 하게 되는지 첫 문단에서 간단히 말한다.
- A/B 중 무엇이 개선안인지, 어느 쪽이 더 오래 걸릴 것인지 과업 전에 직접 말하지 않는다.
- 낯선 용어가 남아 있으면 아래쪽 `용어 설명 보기`에서 뜻을 확인하게 한다.

### 5.3 과업 준비 화면

- `이번 요청`에는 목표 상황만 간단히 적는다.
- 버튼 이름, 탐색 순서, 특정 키 사용법처럼 과업 해결 전략이 될 수 있는 내용은 적지 않는다.
- 단, 과업 종료 방식은 실험 진행 자체에 필요하므로 반드시 안내한다. 특히 과업 종료 버튼을 누르면 종료 확인 대화상자가 열리고, `예`를 눌러야 종료된다는 점은 설명한다.
- 사전 계산 기준, 사용자 유형별 예상 시간, 개선폭은 과업 수행 전에 보이지 않게 한다.
- 두 화면을 모두 수행한 뒤 결과가 한 번에 나온다는 사실은 안내한다.

### 5.4 과업 수행 페이지

- 과업 수행 페이지 안에는 서비스 콘텐츠와 하단 보조 영역만 둔다.
- 비교안 설명, 연구 목적, 예상 시간, 목표 개선폭 같은 실험 운영 문구를 넣지 않는다.
- 실제 사이트처럼 검색, 알림, 보관, 내 정보, 신고 같은 보조 기능을 둘 수 있다.
- 간단한 보조 기능은 가능하면 화면 안 패널, 상태 변경, 조건 입력, 안내 표처럼 실제 기능에 가깝게 반응하게 한다.
- 구현 범위가 큰 보조 기능만 페이지 밖으로 이동하지 않고 `현재 점검 중` 안내를 보여 준다.
- 하단 보조 영역은 기본적으로 `...으로 이동`, `과업 종료` 두 버튼만 둔다. 답변 제출이 필요한 과업은 별도 제출 버튼을 만들지 않고, 하단 보조 영역에서 답변 선택과 과업 종료를 한 번에 처리한다.
- 과업 수행 중에는 `성공했습니다`, `다른 항목을 선택했습니다`, `아직 완료되지 않았습니다`처럼 정답 여부를 알려 주는 문구를 노출하지 않는다.
- 과업 종료 버튼을 누르면 먼저 종료 여부만 묻고, `예`를 누른 뒤 열리는 기록 저장 대화상자에서만 최종 수행 결과 문구를 보여 준다.
- 수행 중 서비스 피드백은 예약 완료 화면, 답글 목록, 토스트 알림처럼 실제 서비스에서 있을 법한 표현으로 제공하고, 정답 여부는 말하지 않는다.
- 실험 번호, 세션 ID, 내부 실행 ID처럼 사용자가 판단할 필요가 없는 식별값은 화면에 노출하지 않는다.
- 사용자가 원하면 `과업 요청 사항을 수행 페이지에서도 보기` 스위치로 과업 수행 페이지 맨 위에도 요청 내용을 표시할 수 있게 한다.

### 5.5 예약 캘린더 화면

- `예약 시간`, `예약 가능한 시간`, `현재 예약 내용`, `조건 적용`처럼 서비스 맥락이 바로 드러나는 표현을 쓴다.
- `예약 시간 안내 보기`, `현재 예약 취소`, `예약 가능 시간으로 바로 이동`처럼 대상과 행동이 함께 드러나는 문구를 쓴다.
- `상담사 소개`, `이용권`, `가격 안내`, `자주 묻는 질문`, `내 상담`처럼 실제 예약 서비스에서 쓰는 메뉴명은 그대로 쓰되, 누르면 화면 안에서 내용을 확인할 수 있게 한다.

### 5.6 댓글 목록 화면

- `댓글 목록`, `답글`, `댓글 정보 보기`, `도움이 돼요`, `정렬 기준`처럼 실제 댓글 화면에서 자주 쓰는 표현을 쓴다.
- `답변 선택 콤보상자`처럼 기능 설명이 필요한 표현은 위치와 목적을 함께 드러내고, 별도 제출 버튼을 반복하지 않는다.
- 답글 작성자 이름처럼 기억에 의한 답변 노이즈가 생길 수 있는 값은 화면마다 배정된 표시값을 기준으로 처리하되, 사용자에게 무작위 배정 사실을 과업 중 힌트처럼 노출하지 않는다.
- 댓글별 보조 기능은 대상 댓글을 구분할 수 있게 이름을 붙인다.
- 보관, 신고, 알림, 작성 안내, 내 댓글 같은 보조 기능은 실제 커뮤니티에서 볼 법한 패널이나 토스트로 처리한다.
- 답글 작성자 이름은 수행 화면별로 달라질 수 있지만, 화면에는 단순히 실제 작성자처럼 표시하고 무작위 배정 사실을 과업 중에 설명하지 않는다.

### 5.7 검색 결과 목록 화면

- `검색 결과`, `정렬 기준`, `자료 범위`, `미리보기`, `저장`처럼 사용자가 바로 이해할 수 있는 표현을 쓴다.
- 저장이나 바로 열기처럼 결과가 짧게 끝나는 기능은 대화상자보다 토스트 알림으로 표시한다. 미리보기 닫기는 별도 안내 없이 이전 흐름으로 돌아가게 한다.
- `정확도순`, `최신 자료 순`, `제목순`처럼 실제 정렬 결과를 예상할 수 있는 이름을 붙인다.
- `검색 결과로 바로 이동`, `선택한 자료 작업`, `바로 열기`처럼 대상과 동작이 함께 드러나는 문구를 쓴다.
- `검색 기록`, `보관함`, `검색 알림`, `자료 요청`, `자료실 문의`처럼 실제 자료 검색 서비스에서 볼 수 있는 메뉴명은 그대로 쓰고, 화면 안 패널이나 토스트로 반응하게 한다.

### 5.8 결과 화면

- 결과 화면은 두 화면을 모두 수행한 뒤에만 보인다.
- `실제 기록`, `사전 계산 기준`, `초점 이동`, `위치 다시 찾기`, `수행 완료 기록`, `수행 불가능 기록`처럼 한국어 중심 지표명을 쓴다.
- 사용자가 이해하기 어려운 개발자 용어 대신 경험 중심 표현을 우선한다.
- 부득이하게 남는 용어는 괄호 설명이나 용어 설명 카드로 보완한다.

## 6. 용어 설명 제공 방식

SC 3.1.3을 만족하기 위한 실무 방식은 다음 중 하나를 쓴다.

1. 화면 안에 바로 짧게 설명을 붙인다.
2. 같은 페이지 아래쪽에 용어 설명 목록을 둔다.
3. 처음 등장한 위치에서만 설명하고, 이후에는 같은 뜻으로만 반복한다.

현재 이 프로젝트는 서비스 화면 하단에 `용어 설명 보기`를 두는 방식을 기본으로 사용한다.

## 7. 점검 목록

배포 전 아래를 확인한다.

- 첫 화면에서 낯선 영어 표현이 바로 보이지 않는가?
- 버튼과 링크 문구만 읽어도 동작을 알 수 있는가?
- 같은 뜻의 용어가 페이지마다 달라지지 않는가?
- 꼭 남겨야 하는 용어에는 뜻을 확인할 방법이 있는가?
- 과업 수행 전에 사전 시간 계산이나 개선 목표를 노출하지 않았는가?
- 과업 요청이 해결 전략을 지나치게 안내하지 않는가?
- 한국형 웹 접근성 관련 표준 용어와 충돌하지 않는가?

## 8. 메모

- 과거 단계 문서에는 이 가이드 적용 전 표현과 삭제된 서비스 유형이 일부 남아 있을 수 있다.
- 현재 공개 화면과 이후 수정 화면은 이 가이드를 기본 기준으로 삼는다.

## 20단계 추가 용어

| 용어 | 설명 | 사용 기준 |
|---|---|---|
| 검색 기록 | 사용자가 최근에 입력하거나 열어 본 검색어와 자료 기록입니다. | `history`만 쓰지 않는다. |
| 보관함 | 사용자가 저장한 자료를 다시 확인하는 영역입니다. | `folder`, `saved`만 쓰지 않는다. |
| 자료 요청 | 찾는 자료가 없을 때 담당 부서에 보완을 요청하는 기능입니다. | 실제 서버 접수처럼 과장하지 않고 실험용 접수 상태로 표현한다. |
| 검색 알림 | 같은 검색어에 새 자료가 올라왔을 때 알림을 받는 기능입니다. | 알림 설정 상태가 바뀌었는지 분명히 알린다. |
| 자료 범위 | 검색 결과에 포함할 자료 종류입니다. 안내문, 질문답변, 신청 서식처럼 구체적인 보기와 함께 쓴다. | `type filter`라고만 쓰지 않는다. |

## 22단계 추가 용어와 표현 기준

| 용어 | 설명 | 사용 기준 |
|---|---|---|
| 전체 메뉴 | 서비스 상단의 주요 이동 메뉴입니다. | 접힘 상태와 펼침 상태가 버튼 문구와 상태로 구분되게 쓴다. |
| 상담 옵션 | 예약 시간 확정 전에 고르는 상담 주제, 사전 작성란, 알림 방법 같은 선택값입니다. | 실제 상담 예약에 있을 법한 범위만 쓰고, 과업 정답 여부를 암시하지 않는다. |
| 한 주 최대 예약 | 같은 주에 예약할 수 있는 횟수 제한입니다. | 제한 안내는 실제 서비스 정책처럼 표현하고, 성공·실패 힌트로 쓰지 않는다. |
| 저장 옵션 | 자료를 어디에, 어떤 내용을 포함해, 어떤 형식으로 저장할지 정하는 선택값입니다. | `save settings` 같은 영어 대신 저장 위치, 포함할 내용, 저장 형식으로 풀어 쓴다. |
| 검색 조건 설정으로 바로가기 | 검색 조건 영역으로 초점을 옮기는 링크입니다. | 단일 페이지 안에서도 긴 조건·결과 영역을 건너뛰는 목적을 드러내기 위해 쓴다. |
| 답변 제출하고 과업 종료하기 | 답변 선택이 필요한 과업에서 답변 제출과 종료를 한 번에 처리하는 버튼입니다. | 별도 제출 버튼을 반복하지 않고, 하단 운영 영역에서 한 번에 처리한다. |


## 23단계 추가 용어와 표현 기준

| 용어 | 설명 | 사용 기준 |
|---|---|---|
| 과업 수행 페이지 | 새 탭으로 열리는 실제 A/B 서비스 조작 화면입니다. | 탭 영역으로 오해될 수 있는 표현을 피한다. 버튼에는 `새 탭 열림`을 함께 적는다. |
| 사전 작성란 | 상담 전에 전달할 내용을 적는 편집창입니다. | 과업에서 비워 두라고 한 경우 선택 옵션처럼 보이지 않게 편집창으로 제공한다. |
| 전체 메뉴 접힘 | 서비스 상단 메뉴가 접힌 상태입니다. | 개선안에서는 접힌 메뉴가 초점 이동 대상에서 빠져야 한다. 조작 부담 비교안에서만 접힌 항목이 초점을 받는 문제를 재현한다. |
