# 14단계 보고서: 서비스 범위 축소, 순서 무작위화, 과업 종료 방식 정비

## 1. 작업 배경

최근 정리 자료는 과도한 키보드·스위치 조작 문제의 핵심을 `키보드로 조작 가능하냐`가 아니라 목표 도달까지 감당해야 하는 조작 비용으로 설명한다. 또한 현재 프로토타입이 7개 서비스 유형, A → B 고정 순서, 수행 탭 첫 조작 계측, 숨김 탭 시간 제외 구조를 갖춘 상태라고 정리했다.

이번 단계에서는 실험을 더 작고 설득력 있게 운영하기 위해 공개 서비스 유형을 3개로 줄이고, A/B 순서가 결과에 영향을 주지 않도록 무작위화했다. 또 사용자가 과업 성공을 자동 판정받는 대신, 스스로 수행을 끝냈다고 판단한 시점에 `과업 종료` 버튼을 눌러 기록을 저장하도록 바꿨다.

## 2. 공개 서비스 범위 변경

### 2.1 남긴 서비스

이번 공개 범위는 다음 3개로 정리했다.

1. 예약 캘린더
2. 댓글 목록
3. 검색 결과 목록

이 세 유형은 반복되는 시간·댓글·자료 항목을 지나야 하고, 보조 동작이 항목마다 반복될 때 조작 비용이 빠르게 늘어나는 구조를 직접 보여 주기 좋다.

### 2.2 제거한 서비스

다음 4개 서비스는 현재 공개 범위에서 제외했다.

- 상품 옵션 선택
- 설정 화면
- 검색 세부 조건
- 신청·결제 흐름

관련 HTML, 앱 스크립트, 시나리오, 과업, 사전 계산 그래프, 사전 계산 결과 파일도 함께 제거했다. 이전 단계 문서는 구현 이력을 위해 남겼지만, 현재 공개 범위는 README와 수동 점검표를 기준으로 한다.

## 3. A/B 진행 방식 변경

### 3.1 기존 방식

이전 단계에서는 모든 서비스에서 비교안 A가 먼저 나오고 B가 나중에 나오는 방식이었다. 이 방식은 첫 진입 조건을 고정하는 장점이 있지만, 학습 효과와 피로 효과가 특정 비교안에 몰릴 수 있다.

### 3.2 변경 방식

`getDefaultConditionOrder()`를 수정해 새 실험을 시작할 때마다 다음 둘 중 하나가 선택되도록 했다.

- A → B
- B → A

이 순서는 각 서비스의 실험 상태 안에 저장되어, 해당 서비스 수행 중에는 같은 순서로 유지된다.

### 3.3 결과 표시 방식

중간 결과 노출을 줄이기 위해 과업 하나 또는 첫 번째 화면을 끝낸 직후에는 결과 화면을 보여 주지 않는다. 다음 과업이나 다음 화면으로 바로 이어지고, 두 화면의 모든 과업을 마친 뒤에만 최종 결과를 한 번에 표시한다.

## 4. 과업 안내 축소

### 4.1 제거한 정보

과업 준비 화면에서 다음 정보를 제거했다.

- 사전 계산 기준표
- 사용자 유형별 예상 시간
- 비교안별 목표 또는 개선폭
- 현재 상태와 목표 상태를 지나치게 구조화한 설명
- 특정 버튼이나 탐색 순서를 알려 주는 세부 단계

### 4.2 남긴 정보

실험 진행에 꼭 필요한 정보만 남겼다.

- 이번 요청
- 현재 몇 번째 화면인지
- 현재 몇 번째 과업인지
- 새 탭에서 수행한다는 사실
- 요청을 다시 확인하려면 메인 창으로 돌아오면 된다는 사실
- 과업을 수행했거나 수행 불가능하다고 판단하면 `과업 종료` 버튼을 누른다는 사실
- 결과는 두 화면을 모두 끝낸 뒤 한 번에 나온다는 사실

## 5. 과업 종료 방식 변경

### 5.1 기존 방식

이전 구조에서는 목표 행동이 감지되면 과업이 자동으로 끝나고, 완료 안내 대화상자를 확인하면 수행 탭이 닫혔다.

### 5.2 변경 방식

이제 목표 행동이 감지되어도 과업이 자동 종료되지 않는다.

1. 사용자가 서비스 화면에서 목표 행동을 수행한다.
2. 서비스 화면 안의 최종 확인 대화상자가 열린다.
3. 사용자가 대화상자의 `확인` 버튼을 누른다.
4. 사용자가 수행 탭 하단의 `과업 종료` 버튼을 누른다.
5. 기록 저장 대화상자가 열린다.
6. 사용자가 `확인`을 누르면 수행 탭이 닫히고 메인 창으로 결과가 전달된다.

목표 행동을 수행하지 않았거나 최종 확인 대화상자를 확인하지 않은 상태에서 `과업 종료` 버튼을 누르면 `수행 불가능 기록`으로 저장된다.

## 6. 실제 서비스 느낌 보강

수행 탭이 너무 실험용 가짜 페이지처럼 보이지 않도록 세 서비스에 보조 기능을 추가했다.

### 6.1 예약 캘린더

- `온마음 상담` 서비스 머리글
- 상담 서비스 검색 입력창
- 알림, 내 상담 버튼
- 상담 안내, 이용 내역, 고객센터 링크

### 6.2 댓글 목록

- `마음돌봄 커뮤니티` 서비스 머리글
- 게시글 요약
- 게시글 보관, 신고 버튼
- 새 글, 내 활동, 알림 링크

### 6.3 검색 결과 목록

- `상담 지원 자료실` 서비스 머리글
- 자료 검색 입력창
- 검색 알림, 보관함 버튼
- 자료실 홈, 최근 자료, 이용 안내 링크

실제로 구현하지 않는 기능은 페이지 이동 대신 `현재 점검 중` 안내를 띄우도록 했다.

## 7. 공통 코드 변경

### 7.1 `data/service-registry.js`

공개 서비스 목록을 3개로 줄였다.

### 7.2 `data/benchmark-manifest.js`

사전 계산 생성 대상을 예약 캘린더, 댓글 목록, 검색 결과 목록으로 줄였다.

### 7.3 `lib/utils.js`

- A/B 순서 무작위화
- 수행 탭 하단 `과업 종료` 버튼 렌더링
- 기록 저장 완료 대화상자 문구 정리
- 점검 중 안내 렌더링 함수 추가

### 7.4 `lib/service-shell.js`

- 서비스 소개 문구를 중립적으로 수정
- 최종 비교 카드에 수행 완료 기록과 수행 불가능 기록을 표시

### 7.5 서비스 앱 파일

- `app.js`
- `comments-app.js`
- `search-app.js`

세 파일 모두 다음 흐름으로 맞췄다.

- 과업 성공 자동 종료 제거
- 최종 확인 대화상자 확인 여부 저장
- `과업 종료` 버튼으로 기록 저장
- 미완료 상태 종료 시 수행 불가능 기록
- 중간 결과 화면 생략
- 실제 서비스 보조 기능과 점검 중 안내 추가

## 8. 문서 변경

- README를 현재 공개 범위와 새 운영 방식 기준으로 다시 작성했다.

## 9. 검증

다음 문법 검사를 수행했다.

```bash
node --check app.js
node --check comments-app.js
node --check search-app.js
node --check lib/utils.js
node --check lib/service-shell.js
node --check lib/experiment-bridge.js
node --check data/service-registry.js
node --check data/benchmark-manifest.js
node --check scripts/run-benchmark.mjs
```

사전 계산 결과를 다시 생성했다.

```bash
node scripts/run-benchmark.mjs
```

생성 대상은 다음 3개 결과 파일이다.

- `data/benchmark-results-calendar.json`
- `data/benchmark-results-comments.json`
- `data/benchmark-results-search.json`

## 10. 남은 점검

이번 단계는 전체 브라우저 수동 점검 세션이 아니라 구조 변경 단계다. 따라서 다음 수동 점검이 남아 있다.

- 새 실험을 여러 번 시작했을 때 A/B 순서가 실제로 섞여 보이는지 확인
- 수행 탭에서 목표 행동 전 `과업 종료`를 눌렀을 때 수행 불가능 기록으로 저장되는지 확인
- 목표 행동 뒤 최종 확인 대화상자를 확인하고 `과업 종료`를 눌렀을 때 수행 완료로 기록되는지 확인
- 두 화면을 모두 끝내기 전 중간 결과가 노출되지 않는지 확인
- 보조 기능이 페이지 밖으로 이동하지 않고 점검 중 안내를 보여 주는지 확인
