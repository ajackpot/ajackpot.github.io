# 3단계 보고서

제목: 예약 캘린더 키보드 점검과 초점 흐름 정비

## 1. 이번 단계 목표

이번 단계의 목적은 예약 캘린더 시범 구현을 실제 키보드 사용 흐름 관점에서 한 번 더 다듬는 것이었다. 특히 다음 네 가지를 정리했다.

1. 과업 화면 안에서 `결과로 바로 이동`이 실제로 초점을 옮기도록 보정한다.
2. 가짜 탐색 링크가 페이지 맨 위로 튀는 부작용 없이 시뮬레이션만 수행되도록 정리한다.
3. 비교안 A는 조작 부담이 크되, 키보드로 끝까지 수행 가능한 구조로 정리한다.
4. 사용자 화면 문구에서 남아 있던 `슬롯` 표현을 정리하고, 결과 지표도 이해하기 쉬운 말로 바꾼다.

## 2. 점검 중 확인한 문제

### 2.1 결과 이동 버튼의 실제 초점 이동 누락

`jump-results` 동작은 기존에 `requestFocus('#results-heading')`만 호출하고 즉시 다시 렌더링하지 않았다. 이 구조에서는 버튼을 눌러도 실제 초점이 바로 이동하지 않을 수 있었다.

### 2.2 시뮬레이션용 링크의 불필요한 기본 동작

상단 보조 내비게이션과 `변경 기준 보기`, `이용 안내 보기`는 실제 서버 없이 시뮬레이션만 보여주기 위한 요소인데, `href="#"`의 기본 동작 때문에 클릭이나 엔터 입력 시 화면이 위로 튈 수 있었다.

### 2.3 비교안 A의 부담 표현 방식이 너무 “고장”에 가까웠음

기존 설명은 대화상자 초점 관리가 약해 배경으로 빠지는 것에 가까웠다. 그러나 이번 프로젝트의 목적은 “키보드로 수행 불가”가 아니라 “키보드로는 가능하지만 조작 횟수가 많고, 다시 찾는 비용이 큰 구조”를 보여주는 데 있다. 따라서 비교안 A는 완전히 깨진 상태보다, **대화상자를 닫은 뒤 넓은 제목 근처로 돌아와 다시 찾게 되는 구조**로 정리하는 편이 더 적절했다.

### 2.4 과업 문구의 잔여 용어

2단계에서 용어 가이드를 만들었지만, 실제 과업 데이터에는 아직 `슬롯`이라는 표현이 남아 있었다. 사용자는 실험 도중 별도 학습 없이 곧바로 이해해야 하므로 이 표현을 정리할 필요가 있었다.

## 3. 이번에 반영한 수정

### 3.1 즉시 초점 이동 도우미 추가

- `focusElementNow(selector)`를 추가했다.
- 이미 화면에 있는 요소는 즉시 `focus()` 하고,
- 아직 다시 그려져야 하는 경우에만 `requestFocus()`로 넘기도록 했다.

이 함수로 `결과로 바로 이동` 버튼과 유사 동작의 안정성을 높였다.

### 3.2 시뮬레이션용 링크 기본 동작 차단

다음 링크에는 `data-inert-link="true"`를 붙이고, 클릭 시 기본 동작을 막았다.

- 상단 보조 내비게이션 링크
- `변경 기준 보기`
- `이용 안내 보기`

이렇게 해서 화면은 그럴듯한 시뮬레이션으로 남기되, 실험 흐름에는 불필요한 맥락 전환이 생기지 않게 했다.

### 3.3 대화상자 초점 흐름 정비

#### 공통

- 대화상자가 열리면 내부 요소로 초점이 이동하도록 정리했다.
- `Tab` / `Shift+Tab`은 대화상자 안에 머물도록 초점 순환을 공통 적용했다.
- `Escape`로 닫기 동작을 유지했다.
- `aria-describedby`를 단순 설명 문장에 연결했다.

#### 비교안 A

- 대화상자를 열면 제목에 먼저 초점을 둔다.
- 닫기나 확정 뒤에는 **방금 조작하던 예약 시간으로 정확히 복귀하지 않고** `예약 가능 시간` 제목 근처로 돌아오게 했다.
- 즉, 고장은 아니지만 다시 맥락을 잡아야 하는 구조로 만들었다.

#### 비교안 B

- 대화상자를 열면 주 동작 버튼에 바로 초점을 둔다.
- 닫으면 방금 열었던 예약 시간으로 되돌아간다.
- 예약을 바꾸거나 취소한 뒤에는 `현재 예약 내용`으로 이동해 다음 단계를 바로 잇도록 유지했다.

### 3.4 사용자 경험 중심 지표 추가

실험 결과 요약에 `대화상자 바깥으로 초점 이탈` 대신, 이번 단계부터 `위치 다시 찾기`를 주요 지표로 사용했다.

의미는 다음과 같다.

- 대화상자를 닫거나 확정한 뒤
- 방금 있던 자리로 정확히 돌아가지 못하고
- 결과 제목이나 넓은 영역부터 다시 찾아야 하는 횟수

이를 위해 로거에 `contextResets` 값을 추가했다.

### 3.5 문구 정리

다음 사용자 표시 문구를 정리했다.

- `슬롯` → `예약 시간`
- `자세히 보기` → `시간 안내 보기`
- 비교안 A 설명에서 `초점이 사라진다`에 가까운 표현 → `결과 제목 근처부터 다시 찾아야 한다`

### 3.6 시각적 초점 표시 보강

스타일에도 다음을 추가했다.

- 공통 `:focus-visible` 외곽선
- 주요 제목의 `scroll-margin-top`
- 상태 상자의 `role="status"` 및 `aria-atomic="true"`

## 4. 사전 계산 기준 갱신

사전 계산 그래프 설명 문구를 이번 초점 흐름에 맞게 다시 썼고, 결과 파일을 재생성했다.

현재 기준 예상 합계는 다음과 같다.

- 키보드: A 140.4초 / B 58.2초 / 82.2초 감소
- 화면낭독: A 245.5초 / B 98.8초 / 146.7초 감소
- 스위치: A 342.6초 / B 139.2초 / 203.4초 감소

이번 단계는 주로 초점 흐름 설명과 사용자 경험 지표를 정리한 것이므로, 예상 시간 값 자체는 바뀌지 않았다.

## 5. 검증

이번 단계에서 수행한 검증은 다음과 같다.

### 5.1 통과

- `node --check app.js`
- `node --check lib/logger.js`
- `node --check data/tasks-calendar.js`
- `node --check data/benchmark-graphs-calendar.js`
- `node scripts/run-benchmark.mjs`
- 정적 서버에서 `index.html` 응답 확인
- `app.js`, `tasks-calendar.js`, `index.html`, `README.md` 대상 금지 표현(`슬롯`, `skip link`, `modal escape`) 재검색

### 5.2 보류

샌드박스 안의 브라우저 정책 때문에 이번 환경에서는 실제 헤드리스 브라우저를 이용한 끝단 키보드 주행 자동화까지는 완료하지 못했다. 따라서 다음 단계 또는 로컬 환경에서 아래 항목을 추가로 확인하는 것이 좋다.

- 홈 → 서비스 선택 → 과업 시작 전체 탭 흐름
- 비교안 A / B의 대화상자 열기와 닫기
- 화면낭독기에서 대화상자 제목과 설명 읽기 순서

## 7. 다음 단계 제안

다음 단계는 두 갈래 중 하나로 바로 이어질 수 있다.

1. 예약 캘린더를 로컬 브라우저에서 실제 키보드와 화면낭독기로 끝까지 점검해 마감 품질을 높인다.
2. 현재 공통 프레임을 유지한 채 다음 서비스 유형(댓글 목록 또는 상품 옵션 선택) A/B 페이지를 추가한다.
