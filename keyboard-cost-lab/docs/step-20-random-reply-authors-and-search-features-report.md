# 20단계 보고서: 댓글 답글 작성자 무작위화와 검색 결과 목록 현실감 강화

## 배경

19단계에서 댓글 답글 질문은 수행 탭 하단의 답변 선택과 `답변 제출하고 과업 종료하기`로 정리했다. 그러나 A 화면에서 답글 작성자를 이미 확인한 사용자가 B 화면에서는 답글 목록을 열지 않고 같은 답을 제출할 수 있는 노이즈가 남아 있었다. 또한 검색 결과 목록은 저장, 미리보기, 바로 열기 같은 핵심 작업은 실제 서비스처럼 동작하지만, 상단 보조 메뉴와 자료실 기능 일부는 여전히 단순 안내에 가까워 실험용 가짜 페이지처럼 보일 수 있었다.

이번 단계에서는 다음 두 가지를 우선 수정했다.

1. 답글 내용은 고정하되 답글 작성자 이름을 수행 화면별로 무작위 배정한다.
2. 검색 결과 목록의 간단한 보조 기능을 화면 안 패널, 토스트, 상태 변경처럼 동작하게 만든다.

## 1. 댓글 답글 작성자 무작위화

### 1.1 문제

댓글 답글 질문 과업은 답글 작성자를 묻는 객관식 과업이다. A/B 화면의 답글 작성자가 고정되어 있으면, 사용자가 첫 화면에서 답을 기억한 뒤 두 번째 화면에서는 답글 목록에 들어가지 않고 바로 하단 답변을 제출할 수 있다. 이 경우 실제로 두 번째 화면의 댓글 구조를 탐색하지 않았는데도 정답 제출이 가능하므로, A/B 비교 결과에 노이즈가 생긴다.

### 1.2 수정 방식

- 답글 본문과 작성 시간은 고정한다.
- 답글 작성자 이름만 과업·비교안별 실행 상태에 저장되는 무작위 쌍으로 배정한다.
- 하단 답변 선택지와 정답도 실제 화면에 표시된 작성자 이름을 기준으로 생성한다.
- 한 서비스 수행 안에서 같은 과업의 다른 비교안은 가능한 경우 이전 비교안의 정답 작성자와 다른 이름을 쓰도록 했다.
- 사용자가 정답 이름만 맞혀도, 지정된 답글 목록에 들어간 기록이 없으면 성공으로 판정하지 않는다.

### 1.3 구현 내용

`comments-app.js`에 다음 구조를 추가했다.

- `REPLY_AUTHOR_NAME_POOL`: 답글 작성자 후보 이름 목록
- `replyAuthorAssignments`: 각 비교안 실행 상태에 보존되는 답글 작성자 배정값
- `buildReplyAuthorAssignment()`: 지정 과업의 대상 답글 정답과 선택지를 생성
- `ensureReplyAuthorAssignmentForTask()`: 이미 만든 배정은 유지하고, 없으면 생성
- `getRenderedReplyAuthor()`: 답글 목록에 실제 표시할 작성자 이름 반환
- `getCurrentReplyQuestionCorrectValue()`: 현재 화면 기준 정답 반환
- `getCurrentReplyQuestionOptions()`: 현재 화면 기준 선택지 반환

이 값은 수행 탭 시작 정보와 실행 상태 직렬화에 포함되므로, 화면을 다시 렌더링해도 같은 수행 탭 안에서는 답글 작성자 이름이 흔들리지 않는다.

## 2. 검색 결과 목록 현실감 강화

### 2.1 문제

검색 결과 목록은 현재 공개 서비스 3개 중 반복 카드와 보조 동작의 조작 비용을 보여 주는 유형이다. 하지만 상단 자료실 메뉴, 검색 알림, 보관함, 검색 기록, 자료 요청 같은 기능이 단순 안내에 가까우면 실제 사이트 탐색처럼 느껴지기 어렵다.

### 2.2 수정 방식

복잡한 외부 이동이나 서버 저장이 필요한 기능은 만들지 않았지만, 화면 안에서 충분히 실제 서비스처럼 느낄 수 있는 기능은 패널과 상태 변경으로 구현했다.

추가 또는 보강한 기능은 다음과 같다.

- 자료실 홈
- 통합 검색 도움말
- 검색 기록
- 저장한 자료
- 보관함
- 자료 요청
- 이용 안내
- 문의
- 검색 알림
- 검색 실행 결과
- 자료 범위 설명
- 자료 갱신 정보

### 2.3 구현 내용

`search-app.js`에 검색 보조 기능 패널을 추가했다.

- `openSearchFeaturePanel()`: 상단 메뉴와 보조 링크를 화면 안 패널로 연다.
- `closeSearchFeaturePanel()`: 패널을 닫고 원래 조작 지점으로 초점을 돌린다.
- `runSearchQuery()`: 검색어를 적용하고 검색 실행 패널을 표시한다.
- `saveCurrentSearchQuery()`: 현재 검색어를 보관한 상태로 표시한다.
- `toggleSearchAlert()`: 검색 알림 켜기·끄기를 상태로 보존한다.
- `setSearchFilterFromFeature()`: 홈 패널의 빠른 조건 버튼이 조건 선택 영역에 값을 입력하게 한다.

상단 검색 폼은 실제 페이지 이동을 하지 않고 현재 검색 결과 화면 안에서 갱신되며, 화면낭독기가 읽을 수 있는 상태 안내도 함께 갱신한다.

## 3. 수행 중 힌트 방지

이번 단계의 보조 기능과 무작위화는 과업 성공 여부를 알려 주지 않는다.

- 답글 작성자 이름은 화면마다 다를 수 있지만, “맞았다/틀렸다”는 수행 중에 표시하지 않는다.
- 검색 보조 패널은 자료실 기능처럼 동작할 뿐, 어떤 자료가 목표 자료인지 알려 주지 않는다.
- 저장, 공유, 문의, 알림 같은 피드백은 실제 서비스 상태 안내처럼 중립적으로 표시한다.
- 최종 성공·실패 판정은 과업 종료 뒤 결과 저장 대화상자에서만 확인된다.

## 4. 벤치마크 영향

이번 변경은 답글 작성자 이름과 검색 보조 기능의 현실감 보강이 중심이다. 댓글 답변 제출 위치는 19단계와 동일하므로 댓글 목록의 사전 계산 그래프 구조는 바꾸지 않았다. 다만 배포 산출물의 일관성을 위해 `node scripts/run-benchmark.mjs`를 다시 실행했다.

현재 댓글 목록 사전 계산 기준 합계는 다음과 같다.

- 키보드 사용자: A 101.0초 → B 34.1초
- 화면낭독 사용자: A 174.4초 → B 56.7초
- 스위치 사용자: A 247.9초 → B 78.0초

## 5. 점검 스크립트 보강

`scripts/smoke-render-runner.mjs`에서 검색 결과 목록 보조 기능 패널도 렌더링 점검에 포함했다.

점검 대상 검색 패널은 다음이다.

- `home`
- `help`
- `history`
- `saved`
- `folder`
- `request`
- `guide`
- `support`
- `alert`
- `search-run`
- `result-history`

댓글 목록과 검색 결과 목록의 보조 기능 패널은 `현재 점검 중` 문구가 남지 않도록 정적 점검에 포함했다.

## 6. 검증 내용

실행한 검증은 다음과 같다.

```bash
node --check app.js

node --check lib/*.js

node --check scripts/*.mjs
node scripts/run-benchmark.mjs
```

## 7. 남은 수동 점검

브라우저와 실제 화면낭독기 조합에서는 다음을 우선 확인한다.

- 댓글 답글 질문 과업에서 A와 B의 답글 작성자 이름이 과업별로 달라질 수 있는지
- 답글 본문과 작성 시간은 유지되며 작성자 이름만 바뀌는지
- 하단 답변 선택지가 실제 표시된 작성자 이름을 포함하는지
- 지정 답글 목록에 들어가지 않고 답만 제출하면 성공으로 기록되지 않는지
- 검색 결과 목록의 상단 메뉴와 보조 링크가 화면 안 패널로 열리는지
- 검색 알림, 검색어 보관, 자료 요청, 문의 접수, 공유가 토스트 또는 상태 변경으로 반응하는지
- 검색 보조 패널이 목표 자료에 대한 힌트를 주지 않는지
