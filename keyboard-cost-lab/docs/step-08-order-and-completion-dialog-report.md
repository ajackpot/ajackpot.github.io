# 8단계 보고서

## 제목

비교안 순서 고정과 과업 완료 대화상자·자동 닫기 보완

## 이번 단계의 배경

사용자 점검에서 두 가지 문제가 확인되었다.

1. 예약 캘린더에서 비교안 B가 먼저 나타나는 경우가 있어, 다른 서비스 유형과의 흐름 일관성이 깨졌다.
2. 과업 성공 뒤 수행 탭이 자동으로 닫히지 않아, 사용자가 직접 `이 탭 닫기`를 다시 찾아야 했다.

또한 두 번째 문제는 예약 캘린더만의 문제가 아니라 댓글 목록과 상품 옵션 선택에도 같은 공통 완료 흐름이 쓰이고 있어, 세 서비스 전체를 함께 점검할 필요가 있었다.

## 확인 결과

### 1. 비교안 순서

기존 구현은 세 서비스 모두 세션 ID 해시값에 따라 `A → B` 또는 `B → A`가 바뀌는 구조였다. 따라서 예약 캘린더에서만 우연히 B가 먼저 보였던 것이 아니라, 세 서비스 모두 같은 가능성을 갖고 있었다.

### 2. 과업 성공 뒤 탭 종료

기존 구현은 과업 성공 시 결과를 메인 창으로 보내고, 수행 탭 안에서는 `이 탭 닫기` 버튼으로 초점을 이동시키는 수준에서 끝났다. 즉, 사용자가 직접 버튼을 한 번 더 눌러야만 탭이 닫혔다. 이 흐름은 예약 캘린더, 댓글 목록, 상품 옵션 선택 세 서비스 모두 동일했다.

## 이번 단계에서 바꾼 내용

### 1. 비교안 순서를 `A → B`로 고정

- `lib/utils.js`에 `getDefaultConditionOrder()`를 추가했다.
- `app.js`, `comments-app.js`, `product-app.js`의 초기 조건 순서를 모두 같은 공통 함수로 바꿨다.
- 이제 세 서비스 모두 비교안 A가 먼저 나오고, 이후 비교안 B가 나온다.

### 2. 과업 성공 뒤 완료 대화상자 추가

- `lib/utils.js`에 `renderRunnerCompletionDialogHtml()`를 추가했다.
- 세 서비스 모두 과업 성공 시 완료 안내 팝업 대화상자가 열리도록 바꿨다.
- 대화상자에는 `확인` 버튼만 두고, 이 버튼을 누르면 수행 탭이 닫히도록 했다.

### 3. 완료 상태에서의 초점과 상호작용 정리

- 과업 성공 시 기존 서비스 화면은 `inert`와 `aria-hidden="true"`로 비활성화했다.
- 완료 대화상자 안으로 초점을 이동하도록 바꿨다.
- 완료 대화상자가 열린 동안 `Tab` 키가 대화상자 밖으로 빠져나가지 않도록 초점 가두기를 적용했다.
- `Escape` 키로 실수로 닫히지 않도록 완료 대화상자 상태에서는 `Escape` 동작을 막았다.

### 4. 기존 수동 닫기 흐름 제거

- 과업 성공 뒤 `이 탭 닫기` 버튼으로 초점을 이동시키던 흐름을 제거했다.
- 대신 결과 전달 직후 완료 대화상자를 보여 주고, 확인 후 자동 닫기 흐름으로 통일했다.

## 검증

다음 항목을 수행했다.

```bash
node --check lib/utils.js
node --check app.js
node --check comments-app.js
node --check product-app.js
node scripts/run-benchmark.mjs
```

검증 결과:

- 세 서비스의 초기 조건 순서가 모두 고정 공통 함수로 바뀌었다.
- 세 서비스 모두 과업 성공 뒤 완료 대화상자를 렌더링하는 코드가 반영되었다.
- 세 서비스 모두 완료 대화상자의 `확인` 버튼이 수행 탭 닫기 동작으로 연결되었다.
- 벤치마크 스크립트는 정상 실행되었고 결과 파일을 다시 생성했다.

## 남은 수동 점검 권장 사항

브라우저 자동화 대신 아래 항목을 실제 브라우저에서 한 번 확인하는 것이 좋다.

- 예약 캘린더, 댓글 목록, 상품 옵션 선택이 모두 `A → B` 순서로 보이는지
- 과업을 성공했을 때 완료 대화상자가 즉시 뜨는지
- 완료 대화상자의 `확인` 버튼이 실제로 새 탭을 닫는지
- 탭 닫기 실패 시 브라우저 정책에 따라 예외가 있는지

## 요약

이번 단계에서는 조건 순서를 세 서비스 모두 `A → B`로 통일하고, 과업 성공 뒤 확인 대화상자를 거쳐 수행 탭이 자동으로 닫히는 흐름으로 정리했다. 이로써 예약 캘린더에서 관찰된 문제를 포함해 다른 서비스 유형에도 같은 구조적 문제가 남아 있지 않도록 공통 흐름 자체를 보완했다.
