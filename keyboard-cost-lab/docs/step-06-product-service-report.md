# 6단계 구현 보고서

부제: 상품 옵션 선택 서비스 추가와 사전 계산 기준 확장

## 1. 이번 단계 목표

이 단계의 목표는 기존의 예약 캘린더, 댓글 목록에 이어 세 번째 서비스 유형인 `상품 옵션 선택`을 같은 실험 틀 안에 추가하는 것이었다. 메인 창에서 과업을 먼저 읽고, 새 탭에서 실제 수행을 분리하는 운영 방식은 그대로 유지하되, 상품 상세 화면에 맞는 상태 모델과 사전 계산 기준을 새로 만들었다.

## 2. 이번 단계에서 구현한 것

### 2.1 홈 화면 연결

- 홈 화면의 `상품 옵션 선택` 카드를 `구현 완료` 상태로 바꿨다.
- 카드에서 `product.html`로 진입하도록 연결했다.
- 홈 화면 소개 문구와 README에 공개 서비스 목록을 반영했다.

### 2.2 상품 옵션 선택 실험 화면

새 파일:

- `product.html`
- `product-app.js`
- `data/product-scenario.js`
- `data/tasks-product.js`
- `data/benchmark-graphs-product.js`
- `data/benchmark-results-product.json`
- `data/benchmark-results-product.js`

핵심 구조:

- 메인 창에서 과업 확인
- 새 탭에서 비교안 A/B 수행
- 첫 입력부터 실제 계측 시작
- 숨김 탭 시간 제외
- 하단 보조 버튼 영역 계측 제외

### 2.3 상품 시나리오와 과업

실험용 상품은 `여행용 파우치 세트`로 구성했다.

옵션 묶음:

- 색상: 크림, 먹색, 올리브
- 크기: 13형, 15형
- 추가 구성: 기본형, 충전기 주머니 포함, 문서 주머니 포함
- 수량: 1개, 2개, 3개

과업 3개:

1. 크림색 15형과 충전기 주머니 포함 구성 담기
2. 13형 설명을 확인한 뒤 먹색 기본형 담기
3. 올리브 15형 문서 주머니 포함 구성 2개 담기

### 2.4 비교안 A/B 구조 차이

비교안 A:

- 상단 보조 링크와 안내 링크를 먼저 지난다.
- 각 옵션 줄마다 `선택`, `옵션 설명 보기`가 따로 있어 순차 이동이 길다.
- 설명 대화상자를 닫으면 `옵션 선택` 제목 근처로 돌아가 다시 찾아야 한다.
- 장바구니 영역이 옵션 묶음 뒤에 놓여 있다.

비교안 B:

- `옵션 영역으로 바로 이동` 링크와 버튼으로 첫 진입 부담을 줄인다.
- 각 옵션 묶음은 한 번만 들어간 뒤 방향키로 고른다.
- `선택한 ... 설명 보기` 버튼을 각 묶음 아래에 하나씩 둔다.
- 대화상자를 닫으면 방금 사용한 설명 보기 버튼으로 초점이 돌아온다.
- 현재 선택과 장바구니 영역에 수량과 담기 버튼을 가까이 모아 둔다.

## 3. 사전 계산 기준 추가

`run-benchmark.mjs`를 확장해 상품 옵션 선택용 결과 파일도 함께 생성하도록 바꿨다.

상품 옵션 선택의 기준 예상 합계:

- 키보드 사용자: A 117.9초 → B 42.0초, 75.9초 감소
- 화면낭독 사용자: A 201.1초 → B 73.4초, 127.7초 감소
- 스위치 사용자: A 294.7초 → B 90.0초, 204.7초 감소

## 4. 문구와 점검 문서 반영

- `docs/language-guideline-wcag-3.1.3-kwcag.md`에 `옵션 묶음`, `장바구니`, `수량` 대응표와 상품 옵션 화면 규칙을 추가했다.
- `docs/manual-release-checklist.md`에 상품 옵션 선택 서비스용 수동 점검 항목을 추가했다.
- README에 상품 옵션 선택 진입점, 파일 위치, 설문지 연동 위치를 추가했다.

## 5. 검증 기록

실행한 점검:

- `node --check app.js`
- `node --check comments-app.js`
- `node --check product-app.js`
- `node --check scripts/run-benchmark.mjs`
- `node scripts/run-benchmark.mjs`
- 정적 서버에서 `index.html`, `product.html`, `data/benchmark-results-product.json` 응답 확인

이번 단계에서는 브라우저 자동화 대신 기존 수동 점검표 확장 방식을 유지했다.

## 6. 다음 단계 메모

이제 구조상으로는 `홈 → 서비스 유형 선택 → 메인 창에서 과업 확인 → 새 탭 수행 → 실제 기록/사전 계산 비교` 흐름이 세 서비스 유형에 공통으로 자리 잡았다. 다음 서비스 유형은 같은 방식으로 독립 페이지와 데이터 파일, 사전 계산 그래프, 단계 보고서를 추가하면 된다.
